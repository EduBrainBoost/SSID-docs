# Token Distribution

The SSID token is a **utility and governance token** — not an investment instrument. Its distribution model is designed for ecosystem participation, long-term sustainability, and fair access rather than speculative extraction.

<Aside type="caution">
The SSID token does not represent equity, debt, or any guaranteed financial claim. There is no redemption at par, no guaranteed buyback, and no profit distribution to holders.
</Aside>

## Token Classification

| Property | Value |
|----------|-------|
| **Primary function** | Utility access + governance voting |
| **Investment product?** | No |
| **Redemption at par?** | No |
| **Profit distribution?** | No — `treasury_nonprofit_only` |
| **Custody model** | Non-custodial (user-held keys) |
| **Transfer restrictions** | Peer-to-peer, no intermediary |

## Sink Model

SSID uses a sink model for treasury management:

- System fees flow into the non-profit treasury
- Treasury funds are allocated exclusively to ecosystem operations
- The treasury operates under `treasury_nonprofit_only` constraints
- No profit extraction, dividend, or buyback mechanisms exist

```text
Transaction Fee (3%)
    ├── 2% → Non-Profit Treasury (sink)
    │         ├── Infrastructure
    │         ├── Security Audits
    │         ├── Open-Source Maintenance
    │         └── Compliance
    └── 1% → Developer Rewards (autonomous routing)
```

## Fairness Signals

Beyond the base sink model, SSID can use governance-defined fairness indicators for discretionary or programmatic community allocations.

One example is a **Proof of Fairness Indicator (POFI)** style signal that prioritizes verified impact over capital concentration.

- High contribution and low historical reward can increase priority for ecosystem programs
- Already saturated reward histories can lower priority for discretionary pools
- Such signals do not override governance rights or protocol safety constraints

## Impact-Linked Programs

Where governance approves outcome-based programs, distribution may be linked to verifiable impact rather than capital size alone. Typical examples include:

- Education completion
- Energy savings
- CO2 reduction
- Other attestable public-good metrics

These programs must remain auditable, configurable through governance, and backed by verifiable credentials or equivalent evidence.

## SBT Roles

SSID uses **Soulbound Tokens (SBTs)** for role-based access control. SBTs are non-transferable tokens bound to a DID.

| SBT Role | Purpose | Transferable? |
|----------|---------|---------------|
| `SBT_CONTRIBUTOR` | Marks verified code contributors | No |
| `SBT_AUDITOR` | Grants audit and evidence review access | No |
| `SBT_GOVERNOR` | Enables governance proposal creation | No |
| `SBT_VALIDATOR` | Authorizes node validation | No |
| `SBT_REVIEWER` | Grants PR review authority | No |

### Properties

- **Non-transferable** — SBTs cannot be sold, traded, or transferred
- **DID-bound** — Each SBT is cryptographically linked to a self-sovereign DID
- **Revocable** — Governance can revoke SBTs through the standard proposal process
- **Auditable** — All SBT issuance and revocation is logged in the evidence trail

## No Redemption at Par

<Aside type="danger">
The SSID token has no redemption mechanism:

- No guaranteed exchange rate
- No buyback program
- No par value
- No fixed redemption into fiat, assets, or other tokens
</Aside>

## Governance Participation

Token holders participate in governance through:

1. Proposal voting
2. Policy gate approval
3. Budget allocation within non-profit constraints
4. Module-level policy changes

Governance scope and restrictions are defined in the [DAO documentation](/governance/dao).
