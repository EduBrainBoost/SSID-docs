# Verifiable Credentials Lifecycle

SSID implements the W3C Verifiable Credentials Data Model v2.0. This document describes the full credential lifecycle: **Issue → Hold → Present → Verify → Revoke**.

## Actors

| Actor | Role | SSID Component |
|-------|------|----------------|
| **Issuer** | Creates and signs credentials | `03_core` + `08_identity_score` |
| **Holder** | Stores and presents credentials | Client wallet (non-custodial) |
| **Verifier** | Validates credential proofs | `14_zero_time_auth` |
| **Revocation Registry** | Tracks credential status | `18_data_layer` + Polygon anchor |

## Phase 1 — Issue

<Steps>

1. **Issuer authenticates** via `did:ssid:polygon` DID
2. **Identity score check** — `08_identity_score` evaluates the subject's trust metrics
3. **Credential construction** — Claims are assembled per schema from `16_codex`
4. **Signing** — Issuer signs with Dilithium (ML-DSA-65) key from `21_post_quantum_crypto`
5. **Anchoring** — Credential hash is anchored on Polygon for tamper evidence
6. **Evidence logging** — Issuance event recorded in `02_audit_logging`
7. **Delivery** — Credential is sent to the Holder via encrypted DIDComm channel

</Steps>

### Credential Structure

```json
{
  "@context": [
    "https://www.w3.org/ns/credentials/v2",
    "https://ssid.example/context/v1"
  ],
  "type": ["VerifiableCredential", "SSIDIdentityCredential"],
  "issuer": "did:ssid:polygon:mainnet:0x7a3b...c4f2",
  "validFrom": "2026-04-01T00:00:00Z",
  "validUntil": "2027-04-01T00:00:00Z",
  "credentialSubject": {
    "id": "did:ssid:polygon:mainnet:0x9e1d...a8b3",
    "identityScore": 87,
    "kycLevel": "enhanced",
    "jurisdiction": "EU"
  },
  "credentialStatus": {
    "id": "https://status.ssid.example/v1/credentials/12345",
    "type": "StatusList2021Entry",
    "statusPurpose": "revocation",
    "statusListIndex": "42",
    "statusListCredential": "https://status.ssid.example/v1/list/1"
  },
  "proof": {
    "type": "DataIntegrityProof",
    "cryptosuite": "dilithium-jcs-2024",
    "verificationMethod": "did:ssid:polygon:mainnet:0x7a3b...c4f2#key-1",
    "proofPurpose": "assertionMethod",
    "created": "2026-04-01T00:00:00Z",
    "proofValue": "<base64url-encoded-signature>"
  }
}
```

<Aside type="tip">
Credentials never contain raw PII. Personal data is represented as hashed claims (SHA3-256) or zero-knowledge proofs. The non-custodial principle applies at every stage.
</Aside>

## Phase 2 — Hold

The Holder stores credentials in their **non-custodial wallet**:

- Private keys never leave the Holder's device
- Credentials are encrypted at rest using the Holder's key
- The wallet supports selective disclosure via zero-knowledge proofs
- No server-side storage of credential content

<Aside type="danger">
SSID is strictly non-custodial. The platform never stores, caches, or has access to the Holder's private keys or unencrypted credential data.
</Aside>

## Phase 3 — Present

When a Verifier requests proof:

<Steps>

1. **Verifier sends a Presentation Request** specifying required claims
2. **Holder reviews** the request in their wallet
3. **Selective Disclosure** — Holder creates a Verifiable Presentation with only the requested claims
4. **Zero-Knowledge Proof** (optional) — For predicates like "age >= 18" without revealing the actual value
5. **Signing** — Holder signs the presentation with their DID key
6. **Transmission** — Presentation is sent via encrypted DIDComm channel

</Steps>

### Verifiable Presentation

```json
{
  "@context": ["https://www.w3.org/ns/credentials/v2"],
  "type": ["VerifiablePresentation"],
  "holder": "did:ssid:polygon:mainnet:0x9e1d...a8b3",
  "verifiableCredential": ["<credential>"],
  "proof": {
    "type": "DataIntegrityProof",
    "cryptosuite": "dilithium-jcs-2024",
    "verificationMethod": "did:ssid:polygon:mainnet:0x9e1d...a8b3#key-1",
    "proofPurpose": "authentication",
    "challenge": "<verifier-nonce>",
    "created": "2026-04-01T12:00:00Z",
    "proofValue": "<base64url-encoded-signature>"
  }
}
```

## Phase 4 — Verify

The Verifier validates the presentation through `14_zero_time_auth`:

<Steps>

1. **Resolve Issuer DID** — Fetch the Issuer's DID Document via `did:ssid:polygon`
2. **Verify Issuer Signature** — Validate the credential proof against the Issuer's public key
3. **Resolve Holder DID** — Fetch the Holder's DID Document
4. **Verify Holder Signature** — Validate the presentation proof
5. **Check Revocation Status** — Query the StatusList2021 revocation registry
6. **Check Expiry** — Validate `validFrom` and `validUntil` timestamps
7. **Trust Score Evaluation** — `08_identity_score` provides risk assessment
8. **Return Verification Result** — Accept or reject with detailed evidence

</Steps>

<Aside type="caution">
Verification must complete all 8 steps. A credential that passes signature verification but is revoked must be rejected. Partial verification is never sufficient.
</Aside>

## Phase 5 — Revoke

Credential revocation is handled via the StatusList2021 mechanism:

<Steps>

1. **Issuer authenticates** with their DID
2. **Set revocation bit** in the StatusList2021 bitstring
3. **Anchor update** — New status list hash anchored on Polygon
4. **Evidence logging** — Revocation event recorded in `02_audit_logging` (WORM)
5. **Propagation** — Verifiers pick up the updated status list on next check

</Steps>

<Aside type="danger">
Revocation is irreversible. Once a credential is revoked, it cannot be un-revoked. The Issuer must issue a new credential if needed.
</Aside>
