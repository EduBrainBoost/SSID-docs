SSID enforces a strict no-secrets-in-repo policy through multiple layers of detection and prevention.

## Current Implementation

### Secret Scanning (CI)

| Repo | Tool | Config |
|------|------|--------|
| SSID | Gitleaks v8.21.2 | `23_compliance/gitleaks.toml` |
| SSID-open-core | grep-based patterns | In `gates.yml` secret_scan job |
| SSID-docs | grep-based patterns | In `docs_ci.yml` secret-scan job |

Detected patterns: `PRIVATE` + `.KEY`, `sk_` + `live_`, `ghp_*`, `gho_*`, `password=`, `api_key=`, `-----BEGIN`, JWT tokens, Base64 secrets.

### PII Redaction (EMS Sanitizer)

The `sanitizer.py` module redacts sensitive data before persistence:

| Pattern | Replacement |
|---------|-------------|
| Windows user paths | `C:\Users\<REDACTED>` |
| Unix home paths | `/home/<REDACTED>` |
| API keys / bearer tokens | `<API_KEY_REDACTED>` |
| Long hex strings (40+ chars) | `<HEX_REDACTED>` |
| Email addresses | `<EMAIL_REDACTED>` |
| Environment variables with values | `<ENV_REDACTED>` |

### Evidence Policy

- **Hash-only**: Evidence stores SHA-256 hashes, never raw content
- **Redaction enforced**: Findings without `redacted=true` are rejected by EvidenceStore
- **No raw prompts/responses**: LLM interactions stored as hashes only

### Pre-Flight Prompt Redaction

The AI Gateway redacts secrets/PII from prompts **before** sending to the LLM:
1. Prompt is scanned by `contains_secrets()`
2. If secrets found: `sanitize_text()` replaces them with placeholders
3. Original prompt hash logged for audit (hash-only, no raw text)

## Planned Enhancements

### HashiCorp Vault Transit

- **Encryption as a service**: Data encrypted without storing keys in application
- **Signing/Hashing/HMAC**: Integrated cryptographic operations
- **Key rotation**: Automated, auditable key lifecycle

### Cloud KMS/HSM

- **AWS KMS** or **Azure Key Vault** for production key management
- **HSM-backed keys** for highest security (FIPS 140-2 Level 3)
- **Audit trail**: All key operations logged via CloudTrail / Azure Monitor

### Encryption at Rest

Currently: plaintext JSON/JSONL on filesystem (acceptable for local development).
Target: Encrypted volumes or Vault Transit for all evidence and state data.
