# DAO Governance

SSID governance is implemented as a **Decentralized Autonomous Organization (DAO)** with strictly scoped authority. The DAO governs system parameters and policy; it does not create a custodial control layer over user funds.

## Governance Scope

| Area | Examples |
|------|----------|
| **System parameters** | Fee thresholds, rate limits, timeout values |
| **Policy gates** | Adding or removing required review gates |
| **Module policies** | Per-root access control, contribution requirements |
| **Budget allocation** | Treasury spending within non-profit constraints |
| **SBT issuance** | Granting and revoking Soulbound Token roles |
| **Protocol upgrades** | Smart contract and governance process upgrades |

## Governance Roles

| Role | Scope | Responsibilities |
|------|-------|------------------|
| **Owner (Per Shard)** | Single shard | Development, quality, release readiness |
| **Architecture Board** | Cross-root | Strategic decisions, API contracts, breaking changes |
| **Compliance Team** | Org-wide | Regulatory oversight, policy enforcement, audit trail |
| **Security Team** | Org-wide | Vulnerability management, incident response, pentesting |
| **DAO / Community** | Protocol-wide | Non-critical governance, treasury allocation, grants |

## Non-Votable Parameters

<Aside type="danger">
The following parameters are immutable and cannot be changed by governance vote:

- `profit_distribution` — No profit distribution mechanism may be introduced
- `redemption` — No token redemption at par may be created
- `treasury_extraction` — No extraction of treasury funds for profit
- `fee_increase_beyond_cap` — Fee percentages cannot exceed hard-coded caps
- `root_structure` — The 24-root structure is locked
- `evidence_deletion` — WORM evidence logs cannot be deleted or modified
- `custody_model` — The non-custodial design cannot be changed to custodial
</Aside>

## Change Classification

Every proposed change is classified using a governance priority framework:

| Priority | Meaning | Example |
|----------|---------|---------|
| **MUST** | Regulatory requirement or security critical | eIDAS compliance, CVE patch |
| **SHOULD** | Best practice, strong recommendation | Logging standardization |
| **COULD** | Nice-to-have, lower priority | Dashboard cosmetic improvement |
| **WOULD** | Future consideration, not currently planned | Experimental DID method support |

## Proposal Lifecycle

```text
1. DRAFT     → Proposer creates proposal
2. REVIEW    → Community review period
3. VOTING    → Token-weighted voting period
4. TIMELOCK  → Mandatory delay before execution
5. EXECUTION → Proposal is executed on-chain
```

<Aside type="note">
The review period ensures that proposals receive community scrutiny before voting begins. Emergency processes may exist, but they operate under stricter thresholds and narrower scope.
</Aside>

## Treasury Rules

The DAO treasury operates under strict non-profit constraints:

| Rule | Enforcement |
|------|-------------|
| No profit extraction | Smart contract invariant |
| Spending requires vote | Governance approval required |
| Operational use only | Infrastructure, audits, maintenance, compliance |
| Transparent accounting | Treasury movements are auditable |
| No investment activity | No yield, lending, or trading deployment |

## Review Structure

| Layer | Role | Review cadence |
|-------|------|----------------|
| 1 | Lead Maintainer + Tech Lead | Every PR |
| 2 | Senior Compliance Officer | Quarterly |
| 3 | External Compliance Expert Pool | Annual |
| 4 | Board Compliance Committee | Escalation only |

## Governance Restrictions Summary

- **Can** adjust scoped system parameters, approve upgrades, and allocate operational budgets
- **Cannot** introduce profit distribution, change custody model, or modify root structure
- **Cannot** delete evidence logs, bypass policy gates, or override security invariants

This design keeps governance effective without undermining the system’s foundational guarantees.
