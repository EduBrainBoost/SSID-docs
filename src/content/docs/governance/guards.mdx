# Guard System

The guard system enforces bounded autonomy for AI agents operating within the SSID ecosystem. Guards are **pre-flight checks** that intercept tool calls (Read, Write, Edit, Bash) before they execute, blocking or warning on policy violations.

<Aside type="note">
Guards operate at the **EMS level** (not in the SSID repo). They are enforced via Claude Code's `PreToolUse` hook protocol, which receives a JSON payload for each tool invocation and returns a block/allow decision.
</Aside>

## Guard Scopes

Each agent is assigned a guard scope that defines what operations it may perform. There are 9 scopes:

| Scope | Purpose | Key Restrictions |
|-------|---------|-----------------|
| `read-only` | Read-only agents (Planner, Scope Sentry) | Blocks all Write, Edit, and Bash calls |
| `write-scope` | Scoped write agents (Implementer) | Allow/deny pattern matching on file paths; hard-deny on `16_codex/`, `.git/`, `.github/` |
| `bash-allowlist` | Controlled shell access | Binary allowlist (python, git, gh, pytest, ruff, mypy, etc.); blocks shell operators (`\|`, `&&`, `;`, `>`) |
| `ems-only` | EMS-internal agents (Evidence, WORM) | Writes must target SSID-EMS paths only; no SSID repo writes |
| `gate-scope` | Gate execution agents (Auditor) | Restricted binary set for gate tools (pytest, gitleaks, semgrep, opa, ruff, mypy) |
| `ops-scope` | Health-check agents (Ops Runner) | Allowed: where.exe, pwsh, python, git, ssidctl, gh, node, npm, pnpm |
| `board-scope` | Board management agents (PM) | Only `ssidctl board`, `ssidctl calendar`, and `ssidctl memory` commands |
| `pr-scope` | PR integration agents | Blocks writes to numbered root dirs (`01_*` through `24_*`) and protected paths |
| `pr-bash-scope` | PR shell operations | Only `git push`, `git checkout`, `git branch`, `git status`, `git log`, `git diff`, `gh pr`, `gh api` |

## Hard Mode vs. Soft Mode

Guards operate in one of two enforcement modes:

| Mode | Behavior | Exit Code on Block | Use Case |
|------|----------|-------------------|----------|
| **Hard** (default) | Blocks the tool call; agent receives an error | Exit 2 | Production; all automated runs |
| **Soft** | Logs a warning but allows the tool call | Exit 0 | Development; debugging guard rules |

Set the mode via environment variable:

```bash
# Default: hard mode (production)
export SSIDCTL_GUARD_SOFT=0

# Soft mode (development/debugging only)
export SSIDCTL_GUARD_SOFT=1
```

<Aside type="caution">
Soft mode is intended for local development only. CI and automated runs always use hard mode. There is no override mechanism in CI.
</Aside>

## Hook Integration with Claude Code

Guards integrate with Claude Code via the **PreToolUse** hook protocol. When Claude Code invokes a tool, the hook receives a JSON payload:

```json
{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "C:/Users/bibel/Documents/Github/SSID/01_identity/new_file.py",
    "content": "..."
  }
}
```

The guard function evaluates the payload against the agent's assigned scope and returns a `GuardResult`:

```json
{
  "blocked": true,
  "reason": "BLOCKED: path '01_identity/new_file.py' not in allowed scope"
}
```

### Guard Evaluation Order

1. **Blocked patterns check** -- Unconditionally blocked patterns (e.g., `Invoke-WebRequest`, `-EncodedCommand`)
2. **Shell operator check** -- Blocks command chaining (`&&`, `||`, `;`, `|`, `>`, `>>`)
3. **Hard-deny paths** -- `16_codex/`, `.git/`, `.github/` are always blocked for writes
4. **Scope-specific rules** -- Allow/deny patterns, binary allowlists, command prefix checks

## Repo-Separation Guards

7 additional guards enforce the boundary between SSID and SSID-EMS:

| ID | Rule | Enforcement |
|----|------|-------------|
| RS-001 | No SSID root writes outside controlled PR workflow | HARD / STOP |
| RS-002 | No `.claude/` directory in SSID repo | HARD / STOP |
| RS-003 | No files directly in SSID root by EMS | HARD / STOP |
| RS-004 | Runtime artifacts (worktrees, evidence, runs, locks) only in SSID-EMS | HARD / STOP |
| RS-005 | SSID access: read via `git -C`, mutate via worktree + PR | HARD / STOP |
| RS-006 | SAFE-FIX enforced: no force-push, reset, clean without approval | HARD / STOP |
| RS-007 | Root-24-lock: no new root directories, no renaming | HARD / STOP |

## Guard-Gate Matrix

The guard-gate matrix defines which guards map to which gates:

- **PRE_FLIGHT guards** run before any Claude interaction (tool call interception)
- **POST_CLAUDE guards** run after Claude produces output but before commit
- **Gates** run in the verify worktree against the actual SSID repo

Every gate produces a deterministic PASS or FAIL result. A single FAIL blocks the PR from merging.

```bash
# Run all guards
ssidctl gates --all

# Run specific guard
ssidctl gates --guard root_24_lock

# Check guard status
ssidctl gates --status
```
