# 24 Root Modules

This page documents every Root module in the SSID 24x16 matrix architecture.

## Root directory

| # | Root | Description |
|---|------|-------------|
| 01 | `01_ai_layer` | AI/ML governance layer enforcing EU AI Act compliance. Provides drift detection, model safety frameworks, and risk classification for all AI-driven components. |
| 02 | `02_audit_logging` | Append-only audit trails stored in WORM-compliant backends. Covers DORA incident response logging and DAO governance audit records. |
| 03 | `03_core` | Central business logic including domain services, public APIs, and tokenomics schemas. All other Roots depend on contracts defined here. |
| 04 | `04_deployment` | Release management and CI/CD pipeline definitions. Handles infrastructure provisioning, rollout strategies, and environment promotion. |
| 05 | `05_documentation` | Developer guides, user manuals, API reference docs, and onboarding materials. Source of truth for all published documentation artifacts. |
| 06 | `06_data_pipeline` | ETL processes, ML/AI data feeds, and analytics pipelines. Manages data ingestion, transformation, and delivery to downstream consumers. |
| 07 | `07_governance_legal` | DAO governance rules, voting mechanisms, and legal framework definitions. Links regulatory requirements to enforceable policy gates. |
| 08 | `08_identity_score` | Identity scoring algorithms and reputation metrics. Computes trust scores through multi-factor risk assessment across identity dimensions. |
| 09 | `09_meta_identity` | Cross-system identity linking and federation. Enables identity portability across chains, protocols, and external verification systems. |
| 10 | `10_interoperability` | API portability contracts, export/import frameworks, and data federation protocols. Ensures SSID components integrate with external ecosystems. |
| 11 | `11_test_simulation` | Testing frameworks, simulation harnesses, and chaos engineering tooling. Provides deterministic replay and fault injection capabilities. |
| 12 | `12_tooling` | Developer tools, CLI utilities, and SDKs. Includes the dispatcher, code generators, and local development environment setup. |
| 13 | `13_ui_layer` | Dashboards, compliance UIs, and frontend components. Renders identity, scoring, and governance data for end users and administrators. |
| 14 | `14_zero_time_auth` | DID-bound session tokens, passwordless authentication, and zero-trust verification flows. Eliminates credential-based auth entirely. |
| 15 | `15_infra` | Infrastructure definitions, DevOps tooling, and cloud-agnostic deployment manifests. Manages networking, secrets, and environment configuration. |
| 16 | `16_codex` | Specification library containing reference implementations, license fee matrices, and canonical rule definitions. Acts as the system's knowledge base. |
| 17 | `17_observability` | Telemetry collection, metrics aggregation, and monitoring dashboards. Provides health checks, alerting rules, and distributed tracing. |
| 18 | `18_data_layer` | Storage backends, data adapters, and schema management. Abstracts persistence so upper Roots remain storage-agnostic. |
| 19 | `19_adapters` | Identity and verification adapters bridging SSID to external ecosystems. Each adapter implements a standardized connector interface. |
| 20 | `20_foundation` | Universal utilities, security libraries, and tokenomics core primitives. Shared dependency for all other Roots. |
| 21 | `21_post_quantum_crypto` | NIST PQC algorithm implementations and quantum-safe key exchange protocols. Provides the cryptographic substrate for long-term security. |
| 22 | `22_datasets` | Reference data, standards catalogs, and example implementations. Supplies seed data for testing, compliance validation, and onboarding. |
| 23 | `23_compliance` | Global compliance matrix mapping regulations to evidence storage. Includes anti-gaming detection and enforcement logic. |
| 24 | `24_meta_orchestration` | Top-level system orchestration, service registry, and version management. Controls CI gates and cross-Root dependency resolution. |

## Structural properties

<Aside type="note">
Every Root contains exactly 16 Shards, producing 384 Root x Shard reference points across the full matrix. See [Shards & Hybrid Charts](/architecture/shards/) for shard details.
</Aside>

**Create-on-use pattern.** Root directories and their shard subdirectories are generated on demand when first referenced by the dispatcher. Empty Roots carry no runtime cost.

**Universal dependencies.** Three Roots are imported by all others and form the architectural baseline:

| Root | Role |
|------|------|
| `20_foundation` | Shared primitives, crypto, utilities |
| `23_compliance` | Regulatory gates applied to every domain |
| `24_meta_orchestration` | Registry, versioning, CI gate enforcement |
