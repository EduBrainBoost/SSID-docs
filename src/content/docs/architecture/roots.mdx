# Root-24 Architecture

SSID is organized into exactly **24 canonical root modules**. This structure is enforced by the **ROOT-24-LOCK** invariant and cannot be modified without an RFC process.

## ROOT-24-LOCK Invariant

<Aside type="danger">
The ROOT-24-LOCK is an immutable system invariant. No agent, process, or human may:

- **Create** new root-level folders
- **Rename** existing roots
- **Delete** any root
- **Restructure** the root hierarchy

Every write operation is validated against the canonical root list before execution.
</Aside>

The lock is enforced at multiple levels:

1. **Path Guard** — Blocks file system writes outside the 24 roots
2. **Scope Router** — Assigns each agent to specific root(s) at session start
3. **PR Gates** — Rejects pull requests that modify root structure
4. **Evidence Logging** — All write operations are logged with SHA-256 hashes

## Common-MUST Structure

Every root module **must** contain the following directory structure:

```
XX_root_name/
├── module.yaml       # Module manifest (dependencies, version, policy gates)
├── README.md         # Module-level documentation
├── docs/             # Detailed design docs and ADRs
├── src/              # Source code
└── tests/            # Unit and integration tests
```

### module.yaml

The `module.yaml` file declares the module's metadata, dependencies, and policy requirements:

```yaml
name: 03_core
version: 0.1.0
description: Core identity logic
dependencies:
  - 21_post_quantum_crypto
  - 18_data_layer
policy_gates:
  - pr_review
  - evidence_append
  - security_scan
```

<Aside type="tip">
The Common-MUST structure ensures that every root is self-documenting, testable, and auditable — regardless of which team or agent maintains it.
</Aside>

## Source of Truth

The canonical definition of all 24 roots lives in **`16_codex`**. All other systems (agents, CI/CD, documentation) derive their root list from this single source. Changes to `16_codex` require an RFC and multi-party approval.
