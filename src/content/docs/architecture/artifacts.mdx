# Deterministic Artifacts

Every state change in SSID produces a **deterministic artifact** — a verifiable output whose hash can be independently computed from its inputs.

## Artifact Properties

| Property | Description |
|----------|-------------|
| **Hash-linked** | Every artifact carries a SHA-256 hash of its contents |
| **Append-only** | Artifacts are never modified, only superseded |
| **Traceable** | Each artifact references the PR, commit, and policy gates that produced it |
| **Reproducible** | Given the same inputs and toolchain version, the same hash is produced |

## Artifact Types

### Source Artifacts
Generated from source code changes:
- Compiled outputs
- Schema validations
- Test results

### Evidence Artifacts
Generated by the audit system:
- Policy gate check results
- PR review records
- WORM log entries

### Configuration Artifacts
Generated from manifest changes:
- Hybrid chart updates
- Dependency graph snapshots
- Root/Shard intersection manifests

## Artifact Lifecycle

```
Source Change → PR Review → Policy Gate Check → Build → Hash Computation → Evidence Write → Artifact Stored
```

<Aside type="tip">
Artifacts are stored alongside their evidence in the `02_audit_logging` root. The evidence directory uses WORM (Write Once Read Many) semantics.
</Aside>

## Verification

Any artifact can be verified by:

1. Checking out the referenced commit
2. Running the deterministic build
3. Comparing the output hash with the stored artifact hash

```bash
# Verify an artifact
ssid verify --artifact-ref 02_audit_logging/evidence/build/artifact_v1.0.0.json
```

If hashes match, the artifact is authentic and untampered.
