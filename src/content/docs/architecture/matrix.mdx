# 24×16 Matrix Architecture

SSID uses a deterministic **24 Root × 16 Shard** matrix to organize all system components.

## Roots (24 Top-Level Domains)

Each root represents a distinct functional domain:

| # | Root | Purpose |
|---|------|---------|
| 01 | `01_ai_layer` | AI model integration gateway |
| 02 | `02_audit_logging` | WORM-compliant evidence storage |
| 03 | `03_core` | Core identity and DID lifecycle |
| 04 | `04_deployment` | CI/CD, containers, rollouts |
| 05 | `05_documentation` | Docs generation and schemas |
| 06 | `06_data_pipeline` | ETL and stream processing |
| 07 | `07_governance_legal` | Policy gates, voting, legal |
| 08 | `08_identity_score` | Trust scoring and reputation |
| 09 | `09_meta_identity` | Cross-chain identity aggregation |
| 10 | `10_interoperability` | Protocol bridging and standards |
| 11 | `11_test_simulation` | Test framework and simulations |
| 12 | `12_tooling` | CLI, dispatcher, dev tools |
| 13 | `13_ui_layer` | Frontend components and views |
| 14 | `14_zero_time_auth` | Instant authentication flows |
| 15 | `15_infra` | Infrastructure and provisioning |
| 16 | `16_codex` | Knowledge base and rulesets |
| 17 | `17_observability` | Metrics, traces, health checks |
| 18 | `18_data_layer` | Persistence and data access |
| 19 | `19_adapters` | External system connectors |
| 20 | `20_foundation` | Shared primitives and crypto |
| 21 | `21_post_quantum_crypto` | Quantum-resistant cryptography |
| 22 | `22_datasets` | Training and reference data |
| 23 | `23_compliance` | Regulatory mapping and enforcement |
| 24 | `24_meta_orchestration` | Task registry, MAOS dispatcher |

## Shards (16 Cross-Cutting Concerns)

Each shard applies across all roots:

| Shard | Purpose |
|-------|---------|
| `config` | Configuration schemas |
| `schema` | Data models, validation |
| `logic` | Business logic |
| `handlers` | Event/request handlers |
| `middleware` | Processing pipeline |
| `validators` | Input/output validation |
| `transformers` | Data transformation |
| `adapters` | External system adapters |
| `repositories` | Data access layer |
| `services` | Service layer |
| `controllers` | Request orchestration |
| `events` | Event definitions |
| `errors` | Error types and handling |
| `types` | Type definitions |
| `constants` | Static values |
| `utils` | Shared utilities |

## Hybrid Charts / Manifests

Every Root×Shard intersection is tracked by a **hybrid chart** — a deterministic JSON manifest that declares:

- Dependencies between components
- Hash of the current artifact state
- Policy gates required for modification
- Evidence trail references

<Aside type="tip">
The matrix is not just organizational — it is the **enforcement layer** for deterministic builds and audit compliance.
</Aside>

```json
{
  "root": "02_audit_logging",
  "shard": "repositories",
  "artifact_hash": "sha256:a1b2c3...",
  "policy_gates": ["pr_review", "evidence_append"],
  "dependencies": ["20_foundation/services"]
}
```
