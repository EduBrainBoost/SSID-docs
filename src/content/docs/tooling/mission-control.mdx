# Mission Control (EMS)

**Mission Control** is the unified operational dashboard for the SSID ecosystem. It combines a CLI tool (`ssidctl`) with a web-based portal that provides real-time visibility into tasks, runs, governance status, and agent activity.

<Aside type="note">
EMS stands for **External Management System** (the user's private vault/wallet). This tooling page documents the evidence management capabilities within Mission Control. All mutations are event-sourced (YAML snapshot + JSONL append-only log) and evidence-linked.
</Aside>

## Board (Task Tracking)

The Board page provides full task lifecycle management with detail views for each task.

### CLI

```bash
ssidctl board add T-001 "Implement auth flow" --module SSID --priority P0
ssidctl board move T-001 DOING
ssidctl board assign T-001 agent:Agent-IMP
ssidctl board update T-001 --priority P1
ssidctl board delete T-001
ssidctl board list
ssidctl board show T-001
```

### Task Statuses

| Status | Description |
|--------|-------------|
| BACKLOG | Not yet scheduled |
| READY | Scheduled, waiting for assignment |
| DOING | In progress |
| BLOCKED | Waiting on dependency |
| REVIEW | Under review |
| DONE | Completed |
| CANCELLED | Cancelled |

Task detail views show the full event history (every status transition, assignment change, and priority update) as an append-only event log.

## Console (Live Terminal)

The Console page streams live PTY output from running agents. Each agent session is isolated and output is streamed to the browser via Server-Sent Events (SSE).

Key properties:
- **Real-time streaming** via SSE (not polling)
- **Per-agent isolation** with separate output buffers
- **Scrollback** retained for the duration of the session
- **No secrets in output** enforced by the guard system (PII/secret scan runs pre-flight)

## Runs

The Runs page lists all EMS runs with detail views that include:

- **Run metadata**: run ID, timestamp, result (PASS/FAIL/STOP), iteration count
- **Logs**: worklog.jsonl event stream for the run
- **Ledger**: append-only approval records with hash chains
- **Gate results**: per-gate PASS/FAIL outcomes with details
- **Handoff summary**: human-readable report generated at run completion

```bash
ssidctl autopilot list     # List all runs
ssidctl autopilot show APR-20260309T120000  # Show run details
```

## Governance

The Governance page provides a unified view of the guard and gate enforcement layer.

- **12 pre-flight guards** (EMS-internal) with current enabled/disabled state
- **11 external gates** (SSID-repo scripts) with last execution results
- **Guard-gate matrix** showing the relationship between guards and gates
- **Enforcement mode** indicator (hard vs. soft) per guard

See the [Guard System](/governance/guards/) page for full details.

## Live Dashboard (SSE Events)

The Live Dashboard page provides a real-time event feed powered by Server-Sent Events.

Events include:
- Task status transitions (`board.task.moved`)
- Gate execution results (`gate.executed`)
- Agent status changes (`team.agent.status_changed`)
- Autopilot iteration completions (`autopilot.iteration.completed`)
- Evidence writes (`evidence.entry.created`)

The SSE endpoint delivers events as they occur with no polling delay.

## Additional Pages

### Memory Vault

```bash
ssidctl memory add "Architecture Decision" --tags adr,architecture --file ./adr-001.md
ssidctl memory list
ssidctl memory show M-abc123
ssidctl memory search "architecture"   # Fulltext search across title, tags, and content
```

### Calendar / Cron

```bash
ssidctl calendar add daily-gates "@daily" "ssidctl gates --all"
ssidctl calendar list
ssidctl calendar disable daily-gates
ssidctl calendar enable daily-gates
ssidctl calendar run-now daily-gates
ssidctl calendar export   # ICS format
```

### Content Pipeline

```bash
ssidctl content new C-001 "Blog: SSID Architecture" --channel blog --tags architecture,overview
ssidctl content stage C-001 SCRIPT
ssidctl content edit C-001 --title "Updated Title"
ssidctl content attach C-001 ./diagram.png --mime image/png
ssidctl content export
ssidctl content list
ssidctl content show C-001
```

Stages: `IDEA` > `OUTLINE` > `BRIEF` > `SCRIPT` > `ASSETS` > `REVIEW` > `PUBLISH` > `ARCHIVE` > `POSTMORTEM`

### Office Dashboard

```bash
ssidctl office              # Snapshot view
ssidctl office --watch      # Live polling (5s default)
ssidctl office --export-json status.json
```

### Team Registry

```bash
ssidctl team list
ssidctl team show Agent-IMP
ssidctl team set-status Agent-IMP RUNNING
```

## Agent Roles

EMS defines 8 agent roles that operate within the 3-worktree model:

| Role | ID | Responsibility |
|------|----|---------------|
| Assistant | --- | Control plane / orchestrator |
| Planner | Agent-PLN | Scope analysis, plan generation |
| Implementer | Agent-IMP | Patch creation in apply worktree |
| Auditor | Agent-AUD | Gate execution in verify worktree |
| Ops Runner | Agent-OPS | CI parity, environment checks |
| Content Writer | Agent-CNT | Content artifact creation |
| Designer | Agent-DSN | Asset briefings, design prompts |
| Board Manager | Agent-PM | Task board and calendar maintenance |

## Data Model

All modules use **event-sourced storage**:

- **Snapshot** (`.yaml`): Current state, overwritten on each mutation
- **Event log** (`.jsonl`): Append-only record of every mutation with timestamp, actor, and payload

```
state/
  board/tasks.yaml + tasks.jsonl
  content/content.yaml + content.jsonl
  team/team.yaml + team.jsonl
  calendar/cron.yaml + runs.jsonl
  memory/index.jsonl + docs/*.md
```

<Aside type="caution">
EMS state directories live outside the SSID repo. Evidence is written to SSID only via PR after gate validation.
</Aside>
